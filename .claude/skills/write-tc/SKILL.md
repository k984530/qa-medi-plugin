# TC 작성

입력: $ARGUMENTS

---

## 0. 저장 규칙

### 파일 형식
- **JSON (.json)** 형식으로 로컬 저장
- Confluence 또는 Google Sheets 업로드 시 변환

### 저장 경로
```
data/tc/{제품명}_{버전}_{기능명}_tc.json
```

**예시:**
- `data/tc/say_v1.2.0_process_guide_tc.json`
- `data/tc/bay_v1.0.0_stock_management_tc.json`

### 파일명 규칙
- 제품명: 소문자, 공백 없음
- 버전: `v{major}.{minor}.{patch}` 형식
- 기능명: 소문자, 언더스코어 구분
- 접미사: `_tc.json`

### JSON 스키마

```json
{
  "metadata": {
    "product": "SAY",
    "version": "v1.2.0",
    "feature": "상담 프로세스 가이드",
    "created_at": "2026-01-26",
    "updated_at": "2026-01-26",
    "status": "초안",
    "source_scenario": "data/scenarios/say_v1.2.0_process_guide_scenario.md"
  },
  "test_cases": [
    {
      "id": "TC-001",
      "section": "1. 프로세스 유형",
      "depth_2": "WEB-상담에이전트",
      "depth_3": "화면 구성",
      "depth_4": "상담 프로세스 유형",
      "depth_5": "초진 고객인 경우",
      "depth_6": "",
      "depth_7": "",
      "precondition": "상담 시작 화면 진입",
      "action": "초진 고객으로 상담 시작",
      "expected": "초진 프로세스(11단계) 자동 적용됨\n- 단계 순서: 첫인사, 아이스브레이킹, ...",
      "priority": "P2",
      "note": ""
    }
  ]
}
```

### Depth 컬럼 설명

| Depth | 용도 | 예시 |
|-------|------|------|
| depth_2 | 제품/플랫폼 | WEB-상담에이전트, Admin, Mobile |
| depth_3 | 화면/영역 | 화면 구성, 녹음 진행 상태, 장애 발생 |
| depth_4 | UI 요소/테스트항목 | [AI 가이드 등록] 버튼, 상태 드롭다운 |
| depth_5 | 상태/조건 | Default, Tap, 미입력 상태 |
| depth_6 | 세부 분기 | 활성화 선택, 검색 결과 있는 경우 |
| depth_7 | 추가 조건의 동작 | 토글 Tap, 문구 확인 |

### 활용 방법

1. **TC 작성 시**: JSON으로 로컬 저장
2. **검토 후**: Confluence 업로드 또는 Google Sheets 동기화
3. **참조 시**: `/write-test-code`에서 TC JSON 읽어서 자동화 코드 생성

## 1. 기본 규칙

- **시나리오에 명시된 내용만 작성** (추측 금지)
- 기획서/정책서에 없는 내용 추가 금지 (Hover, 색상, 아이콘 등)
- 기존 TC 수정 시에도 시나리오 기준으로 검증 필수
- 모든 TC는 문서 출처를 명시할 수 있어야 함

### 데이터 소스 우선순위
1. **Confluence (기획서/정책서)** - 기본 기준
2. **Figma** - 문구 불일치 발견 시 참고
3. **로컬 JSON** - 참고용

### 문구 충돌 발견 시
- 기획자에게 동기화 요청
- TC에는 "(문구 확인 필요)" 표시
- 로컬 파일과 Confluence 충돌 시 → Confluence 기준

### QA 관점 추가 케이스
- TC에 직접 작성하지 않음
- TC 작성 완료 후 별도 "QA 검토 필요 항목" 목록으로 정리해서 사용자에게 전달

## 2. 필수 포함 TC

| 유형 | 필수 케이스 |
|------|-------------|
| 입력 필드 | 정상 입력, 경계값(최대), 경계값 초과, 빈 값, 형식 오류 |
| 버튼 | 비활성화(필수 미입력), 활성화(필수 입력), 클릭 동작 |
| 화면 전환 | 정상 이동, 저장 확인 Alert, 뒤로가기 |
| 권한 | 권한 있음/없음, URL 직접 접근 |
| 에러 | 네트워크 오류, 중복 데이터, 시간 초과, 권한 없음 |

## 3. 용어 규칙

| 사용 금지 | 대체 용어 |
|-----------|-----------|
| 테이블 | 리스트 |
| 컬럼 | (제거) 또는 "영역" |
| 버튼 텍스트 | (제거) - 버튼명만 표기 |
| 헤더 | 리스트 헤더 또는 제거 |

### 디자인 스펙 사용 금지 (필수)

**TC에는 디자인 구현 상세(픽셀, 색상 코드, 디자인 토큰, 스타일 스펙)를 포함하지 않는다.**
이러한 값은 개발/디자인 영역이며 QA TC에서는 기능적 상태로 표현한다.

| 유형 | 사용 금지 | 올바른 표기 |
|------|-----------|-------------|
| 너비 픽셀값 | w222, w58, w350 등 | (제거) 또는 "고정 너비 영역" |
| 색상 Hex 코드 | #5442FF, #FF4E51, #FBFBFB 등 | "파란색(Main Blue)", "빨간색(Main Red)" 등 색상명으로 단순화 |
| 디자인 토큰명 | Gray 500, Main Blue #5442FF 등 | "비활성화 상태", "활성화 상태" 등 기능적 상태로 변환 |
| 버튼 스타일 스펙 | 아웃라인, Filled, 버튼 위치/스타일 설명 | (제거) - 버튼명과 동작만 표기 |

**예시:**
```
❌ [저장] 버튼 (Filled, #5442FF, w222)
❌ 비활성화 상태 (Gray 500, #CBCBCB)
❌ 에러 문구 빨간색(#FF4E51)으로 노출됨

✅ [저장] 버튼이 활성화 상태로 노출됨
✅ 버튼이 비활성화 상태로 노출됨
✅ 에러 문구 빨간색으로 노출됨
```

## 4. TC 배치 및 Depth 구조 규칙

### 4.0 TC 배치 순서 (필수)

**핵심 원칙: 메뉴/화면 UI 순서로 배치한다. 시나리오 작성 순서가 아니다.**

TC는 실제 Admin 화면의 UI 요소 배치 순서를 따른다:

```
1 Depth: JIRA 컴포넌트 (메뉴 대분류)
  2 Depth: 화면명 (메뉴 진입 경로)
    3 Depth: 화면 영역 (상→하, 좌→우 순서)
      4 Depth: UI 요소 (해당 영역 내 배치 순서)
```

**화면 영역 순서:**
```
① 간편 메뉴 (최상단 헤더) → ② 검색 영역 → ③ 버튼 영역 (리스트 상단)
→ ④ 리스트 영역 (정렬, 페이지네이션) → ⑤ 등록/수정 화면 → ⑥ Footer
```

**메뉴 배치 순서 예시 (참고 TC 기반):**
```
제품
 ├── 제품 관리
 │    ├── 간편 메뉴 → [제품] 선택 → [제품 관리] 선택
 │    ├── 검색 영역 → 구분 드롭다운 → 종류 드롭다운 → 제조사 드롭다운 → 제품명 검색
 │    ├── [엑셀 다운로드] / [제품 등록] 버튼
 │    ├── 리스트 영역 → 정렬 → 페이지네이션 → [일괄수정] → [수정] → [일괄삭제] → [삭제]
 │    ├── 제품 등록 → 뒤로가기 → 입력 영역 → [저장] 버튼
 │    └── 제품 수정 → 뒤로가기 → 수정 영역 → [저장] / [삭제] 버튼
 ├── 카테고리 관리 (메뉴 순서 2)
 └── 발주처 관리 (메뉴 순서 3)
```

### 4.0.1 예외/에러 케이스 배치 규칙 (필수)

**핵심 원칙: 예외 케이스를 별도 섹션으로 분리하지 않는다. 해당 UI 요소의 하위 조건으로 인라인 배치한다.**

```
✅ 올바른 배치 (인라인):
[일괄삭제] 버튼
  ├── 선택된 체크박스가 없는 경우 → 토스트 "삭제할 제품을 선택해주세요"
  ├── 선택된 체크박스가 있는 경우
  │    ├── 1개인 경우 → Alert "제품을 삭제하시겠습니까?"
  │    │    ├── [취소] 버튼 → 화면 유지
  │    │    └── [삭제] 버튼 → 삭제 완료 토스트
  │    └── n개인 경우 → Alert "n개의 제품을 삭제하시겠습니까?"
  ├── 현재 발주 중인 제품 포함 → 토스트 "발주 중인 제품은 삭제할 수 없습니다"
  └── 재고 남아있는 제품 포함 → 토스트 "재고가 남아 있어 삭제할 수 없습니다"

❌ 잘못된 배치 (분리):
[일괄삭제] 버튼 → 정상 동작만 작성
...
예외 케이스 (별도 섹션)         ← 분리 금지
  ├── 발주 중인 제품 삭제 시
  └── 재고 남은 제품 삭제 시
```

**규칙:**
- 에러 토스트, 삭제 불가, 권한 없음 등 예외는 해당 동작의 **조건 분기**로 배치
- 같은 버튼의 정상/예외가 하나의 Depth 트리 안에서 표현
- 뒤로가기, 변경사항 미저장 Alert 등도 해당 화면의 UI 요소 하위에 배치

### Depth 컬럼별 역할

- **4 Depth (테스트항목)**: UI 요소명 (예: [AI 가이드 등록] 버튼, 상태 드롭다운)
- **5 Depth (세부케이스)**: 상태/조건 (예: Default, Tap)
- **6 Depth (추가조건)**: 세부 분기 (예: 활성화 선택, 비활성화 선택)
- **7 Depth (추가조건의 동작/결과)**: 6 Depth 조건에서의 동작 (예: 토글 Tap, 문구 확인)

**순서 규칙:**
- 없는 경우 → 있는 경우 순서로 작성
- Default → Tap 순서 (Hover는 시나리오에 명시된 경우만)

### 4.1 계층 구조 원칙 (중요)

**핵심 원칙: 조건 하위에 해당 동작/결과 배치**
- 특정 조건에서만 발생하는 동작은 해당 조건의 하위 Depth에 배치
- 동일한 동작이라도 조건에 따라 결과가 다르면 각 조건 하위에 별도 배치
- 모든 조건에 공통인 동작은 상위 레벨에 배치

**패턴 1: 검색 결과 있음/없음**
```
5 Depth: 텍스트 입력
  6 Depth: 검색 결과 있는 경우
    7 Depth: 검색어 변경/삭제 시
  6 Depth: 검색 결과 없는 경우
    7 Depth: 문구 확인
```

**패턴 2: 버튼 활성화 조건**
```
4 Depth: [초기화] 버튼
5 Depth: 검색 필드 미입력 상태
5 Depth: 검색 필드 입력 상태
  6 Depth: 버튼 Tap
```

**패턴 3: 토글 상태별 동작**
```
4 Depth: 활성화 토글
5 Depth: 토글 상태
  6 Depth: 활성화 상태인 경우
    7 Depth: 토글 Tap → 비활성화로 변경
  6 Depth: 비활성화 상태인 경우
    7 Depth: 토글 Tap → 활성화로 변경
5 Depth: 토글 변경 반영
5 Depth: 진행 중 상담 영향
```

**패턴 4: 메뉴 항목 동작**
```
4 Depth: 더보기 [⋮] 메뉴
5 Depth: Default
5 Depth: [⋮] 버튼 Tap
5 Depth: [상세] 메뉴
  6 Depth: Hover
  6 Depth: Tap
```

**패턴 5: 버튼 활성화/비활성화 조건 분리**
```
4 Depth: [완료] 버튼
5 Depth: 비활성화 조건
  6 Depth: 가이드명 미입력
  6 Depth: 가이드 내용 미입력
5 Depth: 활성화 조건
  6 Depth: 가이드명 AND 가이드 내용 입력됨
5 Depth: 버튼 Tap
  6 Depth: 성공 시
  6 Depth: 실패 시
```

**패턴 6: Alert 내 버튼 동작**
```
4 Depth: [취소] 버튼
5 Depth: 입력 있음
  6 Depth: Alert 노출
    7 Depth: [취소] Tap
    7 Depth: [확인] Tap
5 Depth: 입력 없음
```

**잘못된 예시 (평면적 구조):**
```
❌ 5 Depth: 검색 결과 있는 경우
❌ 5 Depth: 검색 결과 없는 경우
❌ 5 Depth: "검색 결과가 없습니다" 문구 확인  ← 이건 "없는 경우" 하위여야 함
❌ 5 Depth: 리스트 항목 클릭                 ← 이건 "있는 경우" 하위여야 함
```

## 5. Expected Result 작성 규칙

**핵심 원칙:**
- 기대결과에는 **결과만 작성** (조건/분기는 Depth에서 표현)
- 하나의 기대결과에는 **하나의 결과만** 작성
- 부가 설명이 필요한 경우 줄바꿈 후 `- ` 사용

**문체 규칙:**
- **긍정**: ~됨, ~적용됨, ~표시됨, ~노출됨, ~활성화됨
- **부정**: ~되지 않음, ~불가함, ~없음
- **지양**: ~된다, ~합니다, ~해야 함 (서술형 혼용 금지)

**금지:**
- 기대결과에 "~한 경우", "~일 때" 등 조건 포함
- 하나의 Expected Result에 "노출됨" 2회 이상 사용
- 쉼표(,)로 여러 결과 연결
- 시나리오에 없는 내용 추가

**권장:**
```
❌ "전체 선택 상태로 노출됨, 모든 가이드 표시됨"
✅ "전체 선택 상태로 노출됨
   - 모든 가이드 표시됨"

❌ "드롭다운 목록 노출됨\n- 전체 / 활성화 / 비활성화 항목 노출됨"
✅ "드롭다운 목록 노출됨 (전체 / 활성화 / 비활성화)"
```

**버튼 활성화/비활성화 TC 올바른 구조:**

✅ 올바른 예시:
| 4 Depth | 5 Depth | 6 Depth | Priority | Expected Result |
|---------|---------|---------|----------|-----------------|
| [AI로 가이드 생성] 버튼 | 모든 입력 비어있음 | | P3 | 버튼 비활성화됨 |
| | 하나 이상 입력됨 | AI 텍스트만 입력됨 | P2 | 버튼 활성화됨 |
| | | 파일만 업로드됨 | P2 | 버튼 활성화됨 |

## 6. Expected Result 상세 표현 형식

#### 토스트 팝업
```
토스트 팝업 노출됨
- 토스트 팝업 문구: ✓ {성공 문구}
```
```
토스트 팝업 노출됨
- 토스트 팝업 문구: ! {실패/경고 문구}
```

#### Alert
```
삭제 확인 Alert 노출됨
- Alert 타이틀: {제목}
- Alert 문구: {내용}
- 버튼: [취소] / [확인]
```

#### Placeholder / 가이드 문구
```
"{전체 문구}" 문구 노출됨
```

#### 드롭다운 / 옵션 리스트
```
[상태 선택 ∧] 드롭다운 열림 상태로 변경되며 옵션 리스트 노출됨
- 옵션 리스트: 전체 / 활성 / 비활성
```

| 5 Depth | 6 Depth | Expected Result |
|---------|---------|-----------------|
| 텍스트 입력 | 30자 입력 | 정상 입력됨 |
| | 31자 이상 입력 | 추가 입력 불가함 (최대 30자) |
| | 미입력 | [완료] 버튼 비활성화 상태로 노출됨 |

| 5 Depth | Expected Result |
|---------|-----------------|
| .txt 파일 업로드 | 정상 업로드됨<br>- 파일명 표시됨 |

| 10MB 초과 | 파일 크기 초과 토스트 팝업 노출됨<br>- 토스트 팝업 문구: ! 파일 크기는 최대 10MB까지 업로드 가능합니다. |

#### 버튼 성공/실패
| 5 Depth | 6 Depth | Expected Result |
|---------|---------|-----------------|
| 버튼 Tap | 성공 시 | 등록 완료 토스트 팝업 노출됨<br>- 토스트 팝업 문구: ✓ 등록이 완료되었습니다. |
| | 실패 시 | 등록 실패 토스트 팝업 노출됨<br>- 토스트 팝업 문구: ! 저장에 실패했습니다. 다시 시도해주세요. |

```
등록/수정 시 입력된 최근 업데이트 정보 노출됨
- 노출 형식: YYYY.MM.DD / 수정한 직원명
```

## 7. 상태 표기

| 항목 | 표기 방식 |
|------|-----------|
| 토글 상태 | ON / OFF |
| 토글 색상 | ON: 보라색, OFF: 회색 (시나리오에 명시된 경우만) |
| 버튼 상태 | 활성화 / 비활성화 |

### 활성화/비활성화 Expected Result 표기 규칙 (필수)

**영문 용어(Active, Disable, Disabled, Enable 등)를 사용하지 않고, 한국어 서술형으로 작성한다.**

| 사용 금지 | 올바른 표기 |
|-----------|-------------|
| Active | 버튼이 활성화 상태로 노출됨 |
| Disable / Disabled | 버튼이 비활성화 상태로 노출됨 |
| Enable / Enabled | 버튼이 활성화 상태로 노출됨 |
| Button is active | 버튼이 활성화 상태로 노출됨 |

**예시:**
```
❌ [저장] 버튼 Disabled
❌ [저장] 버튼 Active 상태
❌ 버튼 상태: Disable

✅ [저장] 버튼이 비활성화 상태로 노출됨
✅ [저장] 버튼이 활성화 상태로 노출됨
✅ [완료] 버튼 비활성화 상태로 노출됨
```

**적용 대상:** 버튼, 토글, 입력 필드, 체크박스 등 모든 UI 요소의 활성화/비활성화 상태

## 8. 기타 규칙

### Priority 규칙
- P1, P2, P3만 사용
- 검색/필터 관련 기능: P2
- UI 기본 상태 확인: P3

### 우선순위 기준
| 우선순위 | 기준 | 예시 |
|----------|------|------|
| **P1** | 핵심 기능, 데이터 무결성, 보안 | 저장/삭제, 권한 검증, 중복 검증 |
| **P2** | 주요 기능, 일반 사용 흐름 | 검색, 필터, 상태 변경 |
| **P3** | UI/UX 세부사항, 부가 기능 | placeholder, 글자수 카운트 |

### 옵션 선택 TC 형식
```
4 Depth: 드롭다운명
5 Depth: Tap
6 Depth: (옵션명) 선택
Expected Result: "선택한 옵션 선택 표시되며 단일 선택만 가능함"
```

### 화면 구성 TC
- 요소 나열 시 실제 화면 순서대로 작성
- 예: "타이틀 / 검색 영역 / 리스트 영역 / [등록] 버튼"

### 통합 표기
- 수정일 + 담당자: 같은 행에 표기 (별도 분리 X)

### TC 삭제 시
- 행 전체 삭제 (빈 행 남기지 않음)
- Google Sheets batch_update의 deleteDimension 사용
- 아래에서 위로 삭제해야 인덱스 밀림 방지

### QA 확인 필요 항목
- 기획서/정책서에 정의되지 않은 QA 필요 항목
- Expected Result에 "(확인 필요)" 추가
- 셀 배경색: 연한 노란색 (color 3)

### 시나리오 변경사항 반영
- 삭제된 기능: TC에서 완전 제거
- 기본값 변경: TC 즉시 반영

## 9. 테스트 기법별 커버리지 검토

TC 작성 완료 후, 아래 테스트 기법이 적절히 반영되었는지 리뷰한다.
**시나리오에 근거가 있는 항목만 TC로 추가하며, 근거 없이 추측으로 추가하지 않는다.**

### 테스트 기법 체크리스트

| 기법 | 설명 | 확인 포인트 | 예시 |
|------|------|-------------|------|
| **경계값 테스트** | 경계 근처 값 검증 | 입력 필드 최대/최소, 리스트 최대 표시 개수 | 30자/31자 입력, 최대 5개 표시 |
| **동등 분할** | 유효/무효 그룹 대표값 | 정상/비정상 입력 그룹 분리 | 정상 이메일 / 형식 오류 이메일 |
| **조건 분기** | 조건별 다른 결과 | 개수/상태에 따른 동작 차이 | 0개→미노출, 1개→노출, 2개 이상→통합 노출 |
| **상태 전이** | 상태 변화 흐름 검증 | 토글, 상태 변경, 모드 전환 | 활성→비활성, 조회→수정 모드 |
| **에러 추측** | 경험 기반 오류 예측 | 네트워크 끊김, 중복 데이터, 동시 접근 | 저장 실패, 가이드명 중복 |
| **긍정/부정 테스트** | 정상/비정상 동작 쌍 | 성공/실패 케이스 쌍으로 존재하는지 | 등록 성공 / 등록 실패 |
| **권한 테스트** | 권한별 접근/동작 차이 | 관리자/일반 사용자 분리, URL 직접 접근 | 관리자→메뉴 노출, 직원→미노출 |

### 리뷰 절차

TC 작성 완료 후 아래 순서로 리뷰를 수행한다:

1. **기법별 스캔**: 위 체크리스트의 각 기법을 시나리오에 대입하여 누락 여부 확인
2. **누락 판단**: 시나리오/기획서/정책서에 근거가 있는데 TC에 빠진 항목 식별
3. **추가 제안**: 누락된 TC를 사용자에게 목록으로 제안 (자동 추가하지 않음)
4. **승인 후 추가**: 사용자 승인 시 TC에 반영

### 리뷰 출력 형식

```
## 테스트 기법별 커버리지 리뷰

### 충족됨
- [x] 경계값 테스트: 글자수 제한 (30자/31자), 최대 5개 표시
- [x] 상태 전이: 토글 ON/OFF, 조회→수정 모드 전환

### 추가 제안
| # | 기법 | 위치 | 추가 TC 내용 | 근거 |
|---|------|------|-------------|------|
| 1 | 조건 분기 | 점검 필요 항목 | 0개/2개 이상 동작 분기 | 정책서 4-1 |
| 2 | 에러 추측 | 가이드 등록 | 동시 수정 충돌 | 정책서 6-3 |

### 해당 없음
- 동등 분할: 입력 필드 없음 (대시보드 화면)
```

## 10. 출력 형식

위 **0. 저장 규칙**의 JSON 스키마 참조.

### 간략 형식 (단일 TC)

```json
{
  "id": "TC-{섹션}-{번호}",
  "depth_3": "화면/영역",
  "depth_4": "UI 요소",
  "depth_5": "상태/조건",
  "depth_6": "세부 분기",
  "precondition": "사전조건 (화면, 권한, 데이터)",
  "action": "수행 동작",
  "expected": "기대 결과",
  "priority": "P1/P2/P3",
  "note": "비고 (선택)"
}
```

### Confluence/Google Sheets 업로드

TC JSON을 Confluence 또는 Google Sheets에 업로드할 때는 테이블 형식으로 변환:

| 2 Depth | 3 Depth | 4 Depth | 5 Depth | 6 Depth | Priority | Expected Result |
|---------|---------|---------|---------|---------|----------|-----------------|
| {depth_2} | {depth_3} | {depth_4} | {depth_5} | {depth_6} | {priority} | {expected} |

### Google Sheets 업로드 시 양식 참조

Google Sheets에 TC를 업로드할 때, 아래 엑셀 템플릿의 **구조와 양식**을 참조하여 시트를 구성한다.

- **참조 파일:** `.claude/skills/write-tc/tc_template.csv`, `.claude/skills/write-tc/tc_template2.csv`
- **참조 범위:** 컬럼 구조, 헤더 명칭, 셀 병합 방식, 시트 레이아웃, 표현 형식
- **참조하지 않는 것:** TC 내용 (내용은 반드시 JSON 데이터 기준으로 작성)

#### 참조 항목

| 항목 | 설명 |
|------|------|
| 컬럼 순서/명칭 | 엑셀 템플릿의 헤더 순서와 명칭을 따름 |
| 셀 병합 패턴 | 같은 Depth 값이 연속될 때 병합하는 방식 참조 |
| 시트 레이아웃 | 헤더 행 위치, 데이터 시작 행, 고정 행/열 등 |
| 표현 형식 | 우선순위 표기, 기대결과 줄바꿈, 비고란 활용 등 |
